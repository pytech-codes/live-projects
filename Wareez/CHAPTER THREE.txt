CHAPTER THREE
SYSTEM DESIGN AND IMPLEMENTATION

3.1 Introduction
This chapter provides a comprehensive exposition of the design and implementation processes undertaken in the development of the Event Registration and Participant Tracking System for Wareez. The design phase was grounded in established software engineering principles, user-centered design methodologies, and modern web development best practices. Particular emphasis was placed on creating an intuitive, responsive, and efficient platform capable of streamlining complex event management tasks while ensuring scalability, security, and maintainability. The implementation phase involved the systematic translation of detailed design specifications into a functional web application using cutting-edge frontend technologies, modular programming approaches, and rigorous testing protocols. This chapter meticulously documents the system architecture, user interface and user experience design, database schema, security framework, implementation technologies, modular component breakdown, and comprehensive testing methodologies. The approach aligns with the principles outlined by Pressman (2014) and Sommerville (2016) regarding software design and implementation, ensuring that the system meets both functional and non-functional requirements.

3.2 System Architecture
The system is architected using a robust client–server model that ensures clear separation of concerns, facilitates scalability, and supports future extensibility. This architectural choice reflects industry best practices for web applications and aligns with the Model-View-Controller (MVC) pattern, which promotes modularity and maintainability (Krasner & Pope, 1988). The client side is exclusively developed using HTML5, CSS3, and vanilla JavaScript, providing a responsive, interactive, and accessible user experience without external framework dependencies. The server side, planned for subsequent implementation phases, will be developed using a suitable server-side technology stack (PHP or Node.js) and will handle business logic, data processing, session management, and database communication through a well-defined API layer.

The architecture comprises four distinct but interconnected layers:

3.2.1 Presentation Layer
The Presentation Layer encompasses all user-facing components and interfaces, including the homepage with dynamic navigation, event registration forms, participant tracking dashboards, administrative panels, and user authentication interfaces. This layer is designed with a mobile-first approach, ensuring optimal user experience across devices ranging from desktop computers to tablets and smartphones. The implementation leverages semantic HTML5 elements for accessibility, CSS3 Grid and Flexbox for responsive layouts, and progressive enhancement techniques to ensure functionality regardless of browser capabilities. The design emphasizes clarity, consistency, and cognitive ease of use, adhering to Nielsen's (1994) usability heuristics and WCAG 2.1 accessibility guidelines.

3.2.2 Business Logic Layer
The Business Logic Layer serves as the system's core engine, responsible for executing all operational rules and processing user requests. This layer handles critical functionalities including user authentication and authorization, event creation and management, participant registration and validation, report generation, data analytics, and notification systems. The layer enforces business rules, maintains data consistency, and implements transaction management to ensure data integrity during concurrent operations. The design incorporates principles from Domain-Driven Design (DDD) to ensure that business logic remains isolated and testable (Evans, 2003).

3.2.3 Data Access Layer
The Data Access Layer abstracts all database operations and provides a secure interface between the Business Logic Layer and the persistent storage. This layer implements the Repository pattern and utilizes prepared statements, parameterized queries, and Object-Relational Mapping (ORM) techniques to prevent SQL injection attacks and optimize database performance. The layer includes comprehensive error handling, connection pooling, and caching mechanisms to ensure efficient data retrieval and manipulation even under high load conditions.

3.2.4 Database Layer
The Database Layer stores all application data in a relational database management system (RDBMS), with MySQL or PostgreSQL as the primary candidates for implementation. The database schema is designed according to Third Normal Form (3NF) principles to eliminate data redundancy and ensure referential integrity (Codd, 1970). Tables are properly indexed, and relationships are enforced through foreign key constraints. The layer supports ACID (Atomicity, Consistency, Isolation, Durability) properties to guarantee reliable transaction processing.

The modular architecture supports horizontal and vertical scaling through stateless design principles, load balancing capabilities, and database sharding readiness. Future enhancements such as microservices migration, API gateway implementation, and containerization using Docker are architecturally supported, ensuring the system's longevity and adaptability to evolving technological landscapes.

3.3 System Design

3.3.1 User Interface and User Experience (UI/UX) Design
The UI/UX design philosophy is founded on the principles of simplicity, accessibility, aesthetic appeal, and cognitive efficiency to maximize user adoption and satisfaction. The design process incorporated user research findings, usability testing results, and established design patterns to create an interface that resonates with target users while maintaining professional standards. The system implements a comprehensive design system with standardized typography, color palettes, spacing rules, and interaction patterns to ensure consistency across all components and pages.

Key UI/UX features and their implementation details include:

3.3.1.1 Responsive Navigation System
The navigation system employs a transparent, fixed-position header that maintains visibility during scrolling while minimizing screen real estate usage. The navigation menu utilizes CSS Flexbox for optimal spacing and includes hover states with smooth transitions to enhance interactivity. The Login button is prominently styled with a distinctive background color (#6dbffd) and rounded corners (border-top-left-radius: 20px, border-bottom-right-radius: 20px) to create visual hierarchy. The navigation implements mobile-responsive behavior through CSS media queries, transforming into a hamburger menu on smaller screens to maintain usability.

3.3.1.2 Hero Section Design
The hero section serves as the primary user engagement point, featuring a compelling headline ("Event Registration & Participant Tracking System"), descriptive subtitle, and dual call-to-action buttons. The design employs a hierarchical typographic scale with the headline at 40px font-weight: 900, creating immediate visual impact. A horizontal rule (hr) with controlled opacity (0.2) and width (60%) provides visual separation without overwhelming the content. The call-to-action buttons are strategically positioned and styled with distinct colors (green for "Get Started", blue for "How To") to guide user behavior through visual cues.

3.3.1.3 Feature Highlights Section
This section utilizes a modern gradient background (linear-gradient(45deg, rgb(0, 56, 211) 0 50%, blue 50% 100%)) to create visual interest and emphasize key system capabilities. The content is organized using CSS Grid with two equal columns, each containing an icon (100px × 100px), bold heading, and descriptive text. The icons serve as visual mnemonics, improving information retention and scanability. The section's border-radius (10px) and centered positioning (margin-left: 350px) create a card-like appearance that stands out from the background.

3.3.1.4 Sign-Up Conversion Section
The sign-up section employs a centered layout with decorative horizontal rules positioned absolutely to frame the headline ("Join Us Today"). The email input field features substantial dimensions (width: 340px, height: 26px) with rounded corners and focus states to enhance usability. The sign-up button uses a contrasting green background with absolute positioning to create visual prominence. 3.3.1.5 Visual Consistency and Design System
The design system establishes consistent rules for typography (Lucida Sans font family), color usage (white text on dark backgrounds, blue accents), spacing (standardized padding and margins), and interactive elements (hover states, transitions). CSS custom properties are utilized for maintainability, enabling global theme updates through variable modifications. The system implements consistent border-radius values (20px for buttons, 10px for cards) and opacity levels to create visual harmony.

The design implementation in `css/style.css` follows BEM (Block Element Modifier) naming conventions for CSS classes, ensuring code maintainability and scalability. The stylesheet is organized into logical sections: reset, typography, layout, components, and utilities, following SMACSS (Scalable and Modular Architecture for CSS) principles.

3.3.2 Database Design
The database architecture is designed to support the full spectrum of event management operations while ensuring data integrity, performance, and security. The schema follows relational database principles with normalization to eliminate redundancy and maintain consistency. The design incorporates indexing strategies, constraint definitions, and relationship modeling to optimize query performance and enforce business rules.

Primary database tables and their detailed schemas include:

3.3.2.1 Users Table
The Users table stores comprehensive user information with the following structure:
- user_id (INT, PRIMARY KEY, AUTO_INCREMENT): Unique identifier for each user
- first_name (VARCHAR(50), NOT NULL): User's given name
- last_name (VARCHAR(50), NOT NULL): User's family name
- email (VARCHAR(100), UNIQUE, NOT NULL): User's email address for authentication and communication
- password_hash (VARCHAR(255), NOT NULL): Bcrypt-hashed password for security
- role (ENUM('admin', 'organizer', 'participant'), DEFAULT 'participant'): User's system role
- phone (VARCHAR(20), OPTIONAL): Contact phone number
- created_at (TIMESTAMP, DEFAULT CURRENT_TIMESTAMP): Account creation timestamp
- updated_at (TIMESTAMP, DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP): Last update timestamp
- last_login (TIMESTAMP, NULLABLE): Last successful login timestamp
- is_active (BOOLEAN, DEFAULT TRUE): Account status flag

3.3.2.2 Events Table
The Events table contains comprehensive event information:
- event_id (INT, PRIMARY KEY, AUTO_INCREMENT): Unique event identifier
- title (VARCHAR(200), NOT NULL): Event title
- description (TEXT, NOT NULL): Detailed event description
- start_date (DATETIME, NOT NULL): Event start date and time
- end_date (DATETIME, NOT NULL): Event end date and time
- location (VARCHAR(255), NOT NULL): Physical or virtual event location
- capacity (INT, NOT NULL): Maximum number of participants
- registration_deadline (DATETIME, NOT NULL): Registration closing date
- status (ENUM('draft', 'published', 'ongoing', 'completed', 'cancelled'), DEFAULT 'draft'): Event status
- created_by (INT, FOREIGN KEY REFERENCES users(user_id)): Event creator
- created_at (TIMESTAMP, DEFAULT CURRENT_TIMESTAMP): Event creation timestamp
- updated_at (TIMESTAMP, DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP): Last update timestamp

3.3.2.3 Registrations Table
The Registrations table tracks participant sign-ups:
- registration_id (INT, PRIMARY KEY, AUTO_INCREMENT): Unique registration identifier
- event_id (INT, FOREIGN KEY REFERENCES events(event_id), NOT NULL): Associated event
- user_id (INT, FOREIGN KEY REFERENCES users(user_id), NOT NULL): Registered participant
- registration_date (TIMESTAMP, DEFAULT CURRENT_TIMESTAMP): Registration timestamp
- status (ENUM('pending', 'confirmed', 'cancelled', 'attended'), DEFAULT 'confirmed'): Registration status
- payment_status (ENUM('pending', 'paid', 'refunded'), DEFAULT 'pending'): Payment status
- notes (TEXT, NULLABLE): Additional registration notes

3.3.2.4 Event_Analytics Table
The Event_Analytics table stores performance metrics:
- analytics_id (INT, PRIMARY KEY, AUTO_INCREMENT): Unique analytics entry
- event_id (INT, FOREIGN KEY REFERENCES events(event_id), NOT NULL): Associated event
- metric_type (ENUM('view_count', 'registration_count', 'attendance_rate'), NOT NULL): Metric category
- metric_value (DECIMAL(10,2), NOT NULL): Metric numerical value
- recorded_date (DATE, NOT NULL): Date of metric recording

Relationships are enforced through foreign key constraints with ON DELETE CASCADE behavior to maintain referential integrity. Indexes are strategically placed on frequently queried columns: email (users table), event_id and dates (events table), and composite indexes on (event_id, user_id) in registrations table for optimal join performance.

3.3.3 Security Design
Security is implemented through a multi-layered approach addressing authentication, authorization, data protection, and communication security. The security framework follows OWASP (Open Web Application Security Project) guidelines and incorporates defense-in-depth principles to protect against common web application vulnerabilities.

3.3.3.1 Authentication and Authorization
Authentication implements secure password practices using Bcrypt with a cost factor of 12 for password hashing, providing resistance against brute-force attacks. Passwords are salted uniquely per user to prevent rainbow table attacks. Session management utilizes secure, HTTP-only cookies with SameSite=Strict attribute to prevent cross-site request forgery. Session tokens are generated using cryptographically secure random number generators and have configurable expiration times. Multi-factor authentication (MFA) is architected for future implementation using Time-based One-Time Passwords (TOTP).

Authorization follows the Principle of Least Privilege (PoLP) with role-based access control (RBAC). Three primary roles are defined: administrators (full system access), organizers (event management capabilities), and participants (registration and viewing permissions). Access control lists (ACLs) are implemented at both the endpoint and data level to ensure users can only access authorized resources.

3.3.3.2 Input Validation and Data Sanitization
Input validation is implemented at multiple levels. Client-side validation provides immediate user feedback using HTML5 form attributes (required, pattern, min/max) and JavaScript validation for complex business rules. Server-side validation enforces all constraints regardless of client-side checks, preventing bypass attempts. All user inputs are sanitized using parameterized queries and output encoding to prevent injection attacks. File upload restrictions are implemented including file type validation, size limits, and virus scanning integration.

3.3.3.3 Data Protection and Encryption
Data in transit is protected using TLS 1.3 with Perfect Forward Secrecy (PFS) cipher suites. Sensitive data at rest, including personally identifiable information (PII), is encrypted using AES-256 encryption with key rotation policies. Database connection strings and API credentials are stored using environment variables or secure configuration management systems. Regular security audits and penetration testing are scheduled to identify and address vulnerabilities.

3.3.3.4 Cross-Site Scripting (XSS) and CSRF Protection
XSS prevention is implemented through output encoding using context-appropriate encoding (HTML, JavaScript, CSS, URL). Content Security Policy (CSP) headers are configured to restrict resource loading and inline script execution. CSRF protection is implemented using synchronizer token patterns with double-submit cookie validation for state-changing operations.

3.3.2 Database Design
The database is designed to be relational, normalized, and secure to support efficient data management and retrieval. Primary tables include:
- Users: Stores user records (user_id, first_name, last_name, email, password_hash, role, created_at).
- Events: Contains event information (event_id, title, description, date, location, capacity, created_by).
- Participants: Details of participants (participant_id, user_id, registration_date, event_id).
- Registrations: Tracks registration data (registration_id, event_id, user_id, status, registration_date).

Relationships are enforced through foreign keys to ensure referential integrity. Indexes are applied to frequently queried columns (e.g., email, event_id, registration dates) to optimize performance. Sensitive data such as passwords are hashed using Bcrypt, and the schema supports future scalability.

3.3.3 Security Design
Security is integrated throughout the system to protect user data and prevent unauthorized access:
- Authentication: Login forms implement secure password practices. Passwords are salted and hashed using Bcrypt before storage. Session management employs secure, HTTP-only cookies.
- Input Validation: Both client-side and server-side validation prevent malicious input. HTML5 form attributes and JavaScript validation provide immediate feedback, while server-side checks enforce data integrity.
- Data Encryption: Sensitive data transmissions are encrypted using HTTPS (TLS). Future enhancements may include AES encryption for specific fields.
- Access Control: Role-based access control (RBAC) restricts administrative functions to authorized users only.
- CSRF and XSS Protection: Forms include CSRF tokens, and output encoding prevents cross-site scripting attacks.
